A container is a way to run an application in isolation from the rest of the system, while sharing the same host operating system.

## The Problem with Virtual Machines

When talking about [[01. Why Virtualization|Virtualization]], virtual machines are the first approach. However, they virtualize everything a full OS requires — including hardware emulation, a kernel, and the guest operating system itself. This results in heavy resource usage and slow startup times.

A container, on the other hand, uses the **same kernel** as the host operating system, but isolates the processes, file system, network, and resources used by each application's runtime environment.

![[Pasted image 20260707020548.png]]

| Feature         | Virtual Machine | Container     |
| --------------- | --------------- | ------------- |
| Startup time    | Minutes         | Milliseconds  |
| Size            | GBs             | MBs           |
| Kernel          | Own (guest)     | Shared (host) |
| Isolation level | Full hardware   | Process-level |
| Portability     | Limited         | High          |

## Container Engine

To containerize applications, a **Container Engine** is required. This is the software layer responsible for pulling images, creating containers, managing their lifecycle, and providing the runtime environment.
