## **Why Docker**:

- _Environment Reproducibility_: Everyone one gets same setup.
- _Dependency Management_: Remove OS, Language or library issues.
- _Portability_: as it runs on every machine that contains Docker.
- _Version Control_: Docker files are text = versions.

### Docker Engine

**Docker Engine** is the most widely used container engine. It is an open-source client-server application that enables building, running, and managing containers on a host machine.

It is composed of three main components:

```
┌──────────────────────────────────────────┐
│              Docker CLI                  │  ← UI (docker run, build…)
├──────────────────────────────────────────┤
│            Docker Daemon (dockerd)       │  ← Background service (core logic)
├──────────────────────────────────────────┤
│         containerd + runc                │  ← Low-level runtime (OCI compliant)
└──────────────────────────────────────────┘
```

#### 1. Docker CLI (`docker`)
The command-line interface through which users interact with Docker. Commands like `docker run`, `docker build`, and `docker ps` are sent from the CLI to the daemon via a REST API.

#### 2. Docker Daemon (`dockerd`)
The long-running background process that does the heavy lifting. It:
- Listens for Docker API requests
- Manages Docker objects: **images**, **containers**, **networks**, and **volumes**
- Communicates with `containerd` to actually run containers

#### 3. `containerd` & `runc`
- **`containerd`** is an industry-standard container runtime that manages the complete container lifecycle (image transfer, storage, execution, and supervision).
- **`runc`** is the low-level OCI-compliant runtime that actually spawns and runs containers by interfacing directly with Linux kernel primitives: **namespaces** and **cgroups**.

```
docker CLI  ──REST API──▶  dockerd  ──▶  containerd  ──▶  runc  ──▶  Container
```

#### Key Concepts

| Concept | Description |
|---|---|
| **Image** | A read-only, layered snapshot of a filesystem. The blueprint for a container. |
| **Container** | A running instance of an image. Ephemeral by default. |
| **Dockerfile** | A text file with instructions to build a Docker image. |
| **Registry** | A repository for images (e.g., Docker Hub, GHCR, ECR). |
| **Volume** | Persistent storage that survives container restarts. |
| **Network** | Virtual network that allows containers to communicate. |

#### Linux Kernel Primitives Used

Docker Engine relies on two core Linux kernel features to achieve isolation:

- **Namespaces** — Isolate what a container *can see*: process IDs (PID), network interfaces (NET), filesystems (MNT), hostnames (UTS), users (USER), and IPC.
- **Control Groups (cgroups)** — Limit what a container *can use*: CPU, memory, disk I/O, and network bandwidth.

> [!info] Docker on macOS / Windows
> Because Docker relies on Linux kernel features, on macOS and Windows it runs a **lightweight Linux VM** (via HyperKit / WSL2) behind the scenes. The containers themselves still share a Linux kernel — just the VM's, not the host OS's.

#### Common Docker CLI Commands

```bash
# Build an image from a Dockerfile
docker build -t my-app:1.0 .

# Run a container
docker run -d -p 8080:80 --name my-container my-app:1.0

# List running containers
docker ps

# View logs
docker logs my-container

# Stop and remove
docker stop my-container && docker rm my-container

# List images
docker images

# Pull an image from a registry
docker pull nginx:alpine
```
